# Environment

**What belongs here:** Required env vars, external dependencies, setup notes.
**What does NOT belong here:** Service ports/commands (use `.factory/services.yaml`).

---

## Database

- **PostgreSQL 18** running in Docker container `sub2api-postgres` via OrbStack
- Host: `sub2api-postgres.orb.local` (OrbStack DNS)
- Port: 5432
- User: `sub2api`
- Database: `involute`
- Connection string: see .env file (auto-generated by init.sh)
- Password is extracted at runtime from container env by init.sh
- Local workers should connect through the managed `postgres_proxy` service on `127.0.0.1:4202`; `.factory/init.sh` starts that proxy and rewrites `DATABASE_URL` accordingly
- `packages/server/prisma/.env` is a committed symlink to the repo-root `.env` so Prisma CLI commands resolve the same `DATABASE_URL`

## Environment Variables

- `DATABASE_URL` — Prisma connection string (auto-generated by init.sh)
- `AUTH_TOKEN` — Service token for GraphQL API authentication (written by `.factory/init.sh`; current default: `changeme-set-your-token`)
- `PORT` — API server port (default: 4200)

## Setup Notes

- Run the `init` command from `.factory/services.yaml` (or `.factory/init.sh` directly) before Prisma or validator commands; it verifies Postgres access, starts `postgres_proxy`, updates `.env`, and runs `prisma generate` / `prisma db push` when applicable
- Current Prisma CLI wiring still uses `packages/server/package.json#prisma`, so `prisma generate` and `prisma db push` emit Prisma 6 deprecation warnings; this is not breaking today but will need a `prisma.config.ts` migration before Prisma 7
